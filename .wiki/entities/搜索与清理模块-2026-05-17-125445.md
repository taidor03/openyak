# 搜索与清理模块

知识中心包含两个重要的工具模块：全文搜索引擎 (`search.py`) 和 Wiki 维护清理工具 (`cleanup.py`)。它们分别是页面检索和 Wiki 健康维护的核心。

---

## search.py — 全文搜索引擎

### 设计定位

基于 **Token + 短语匹配** 的轻量全文搜索引擎，从 TypeScript 项目 `nashsu/llm_wiki` 移植而来。不依赖外部数据库或向量存储，直接对 Markdown 文件内容进行内存匹配。

与原版相比做了以下简化：
- 无向量/嵌入搜索（需要外部 LLM）
- 无 RRF 融合（只有单一排名列表）
- 无 Tauri IPC / FileNode 树 — 直接使用 pathlib
- 无图片提取

### 核心流程

```
用户查询 "MCP 配置"
    ↓
tokenize_query(query) → ["mcp", "配置"]  (含 CJK bigram)
    ↓
遍历 <wiki_root>/<category>/*.md，逐文件评分
    ↓
_score_file() 计算匹配分数 (文件名精确 + 标题短语 + 内容短语 + token 匹配)
    ↓
按 score 降序排序 → 返回 Top N SearchResult
```

### 评分权重

| 因素 | 权重 | 说明 |
|---|---|---|
| `FILENAME_EXACT_BONUS` | 200 | 文件名 stem 与查询短语完全一致（最强信号） |
| `PHRASE_IN_TITLE_BONUS` | 50 | 标题中包含完整查询短语 |
| `PHRASE_IN_CONTENT_PER_OCC` | 20 × 出现次数 | 内容中查询短语每次出现加 20 分（上限 10 次 = 200 分） |
| `TITLE_TOKEN_WEIGHT` | 5 / token | 标题命中每个 token 加 5 分 |
| `CONTENT_TOKEN_WEIGHT` | 1 / token | 内容命中每个 token 加 1 分 |

> 精确词法匹配主导排名——用户输入 "attention" 时，名为 `attention.md` 的页面**必定**排名第一。

### 关键函数

#### `tokenize_query(query: str) -> list[str]`

分词查询字符串。处理西方文字和 CJK 文本：
- 西方文字：按空白和标点分割，转小写，过滤停用词
- CJK：额外生成重叠 bigram（`"配置"` → `["配置", "配", "置", "配置"]`）
- 结果去重并保持顺序

```python
tokenize_query("MCP 配置方法")
# → ["mcp", "配置", "配", "置", "方法", "方", "法", "配置方法"]
```

#### `_score_file(file_path, content, tokens, query_phrase, query) -> SearchResult | None`

纯评分函数（无 IO）。对单个文件计算匹配分数：
1. 提取标题（frontmatter `title:` → `# heading` → filename stem）
2. 计算精确匹配信号：文件名精确、标题短语、内容短语出现次数
3. 计算 token 匹配信号：标题 token 数、内容 token 数
4. 若所有信号为零，返回 `None`（不纳入结果）
5. 加权求和得分

#### `search_wiki(wiki_root: str, query: str, max_results: int = 20) -> list[SearchResult]`

主搜索入口。遍历 `wiki_root` 下所有分类子目录中的 `.md` 文件，评分并返回排名结果。

### 数据结构

```python
@dataclass
class SearchResult:
    path: str          # 文件绝对路径
    page_id: str       # 文件名 stem（去掉 .md）
    title: str         # 提取的页面标题
    category: str      # 所属分类（路径倒数第二级目录名）
    snippet: str       # 匹配上下文片段（前后各 ~80 字符）
    title_match: bool  # 标题是否命中
    score: float       # 匹配分数
```

### 搜索片段生成

`_build_snippet(content, query)` 在内容中找到第一个查询匹配位置，提取前后各 `SNIPPET_CONTEXT`（80）字符的上下文片段，超出部分用 `...` 标记。

### CJK 分词策略

对于 CJK 文本（长度 > 2），`tokenize_query` 同时生成：
- 重叠 bigram：每相邻两字组合（`chars[i] + chars[i+1]`）
- 单字：每个字符（过滤停用词）
- 原始 token

这保证了即使查询不完整也能匹配到相关页面。

### 性能说明

- **适用规模**：个人 Wiki（数百 ~ 数千页面）完全胜任
- **瓶颈**：每次搜索全量读取所有 `.md` 文件；万级以上页面建议引入索引
- **CJK 支持**：内置 bigram + 单字分词，无需外部分词器

### 停用词

`STOP_WORDS` 包含中英文常见停用词（"的", "是", "the", "is" 等），分词时自动过滤。

## cleanup.py — Wiki 维护与清理工具

### 设计来源

从 TypeScript 项目 `nashsu/llm_wiki` 的 `src/lib/wiki-cleanup.ts` 移植。专门解决两类真实 Bug：

| Bug | 问题 | 解决方案 |
|---|---|---|
| **Bug A（漏删）** | 用 `includes` 匹配 slug 时，`[[KV Cache]]` 的 human title 形式无法匹配文件 slug `kv-cache` | 改用规范化键值比较（`normalize_wiki_ref_key`） |
| **Bug B（误删）** | 子串 `includes` 导致 `ai.md` 被删除时 `[[OpenAI]]` 也被误删 | 精确匹配 wikilink 目标，非子串匹配 |

### 核心工具函数

#### `normalize_wiki_ref_key(s: str) -> str`

规范化引用键值，使得不同写法指向同一页面：

```python
normalize_wiki_ref_key("KV Cache")    # → "kvcache"
normalize_wiki_ref_key("kv-cache")    # → "kvcache"
normalize_wiki_ref_key("kv_cache")    # → "kvcache"
normalize_wiki_ref_key("wiki/entities/kv-cache.md")  # → "kvcache"
```

处理步骤：去除路径前缀和 `.md` 后缀 → 转小写 → 移除空格/连字符/下划线。

### merge_sections

```python
def merge_sections(existing_body: str, new_sections_text: str) -> str:
    """纯算法章节合并（无需 LLM 调用）
    
    合并规则:
    1. 前言（existing 第一个 ## 之前的内容）始终保留
    2. 新章节标题与现有章节标题匹配（规范化比较）→ 替换该章节内容，但保留原始标题级别
    3. 新章节标题无匹配 → 追加到末尾，使用新章节的标题级别
    4. 现有章节顺序保持不变，被替换的章节留在原位
    """
```

合并示例：
```
existing_body:
# 标题
前言内容

## 第一章
现有内容

## 第二章
现有内容

new_sections_text:
## 第二章（更新版）
新内容

## 第三章（全新）
全新章节

结果:
# 标题
前言内容

## 第一章
现有内容

## 第二章
新内容

### 第三章（全新）
全新章节
```

## 两个模块的协作

```
删除页面 (DELETE /api/wiki/pages/{id})
    │
    ├─→ service.py: delete_page()
    │       ├─→ cleanup.build_deleted_keys()      # 生成删除键集合
    │       ├─→ cleanup.clean_index_listing()      # 清理 index.md
    │       └─→ cleanup.strip_deleted_wikilinks()  # 清理所有页面中的失效链接
    │
    └─→ 返回 cleaned_references 计数

章节合并 (POST /api/wiki/merge)
    │
    └─→ service.py: merge_pages()
            ├─→ cleanup.parse_sections()    # 解析现有/新内容为章节
            └─→ cleanup.merge_sections()    # 算法合并，保留原结构

搜索 (POST /api/wiki/search)
    │
    └─→ search.search_wiki(wiki_root, query)
            ├─→ tokenize_query()            # 分词（含 CJK bigram）
            ├─→ _score_file() × N           # 对每个文件评分
            └─→ 按 score 降序排序 → Top N
```
