# OpenYak 对话交互全链路深度分析报告

> 生成日期: 2026-05-19  
> 范围: 从用户输入开始，经过对话发展逻辑处理、工具调用机制、与LLM模型交互流程，直至对话内容生成与最终显示的完整过程

## 一、系统架构总览

OpenYak 采用 **三层分离架构**：Tauri 桌面壳（Rust）→ Python 后端（FastAPI）→ React 前端（Next.js 15）。核心设计决策（ADR-0001）：Tauri 仅负责壳层关注（窗口、进程管理、原生对话框），所有 AI/Agent/Tool 逻辑驻留在 Python 后端，前端通过 HTTP + SSE 与后端通信。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Tauri Shell (Rust, ~875行 backend.rs)              │
│  backend.rs: Python子进程生命周期 + 看门狗 + 自动重启                    │
│  commands.rs: 11个IPC命令  tray.rs: 系统托盘  menu.rs: 原生菜单        │
│  lib.rs: 插件注册 + 事件循环                                          │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ Tauri IPC (invoke/listen)
                           ▼
┌─────────────────────────────────────────────────────────────────────┐
│              React Frontend (Next.js 15 + React 19, ~50+组件)       │
│  Zustand: 流式瞬时态(chat-store 569行, 25+actions)                    │
│  React Query v5: 服务端缓存(消息/会话, 无限查询+分页)                   │
│  SSEClient: 双模式(EventSource本地 / fetch+ReadableStream远程)         │
│  ProgressiveBuffer: 60ms批量合并delta(防Chromium/WindowServer卡顿)   │
│  hooks/: use-chat, use-sse(837行), use-messages, use-sessions        │
│  components/: chat/, messages/, parts/, artifacts/, interactive/     │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ HTTP + SSE (localhost:{port})
                           ▼
┌─────────────────────────────────────────────────────────────────────┐
│              Python Backend (FastAPI + SQLAlchemy async)             │
│  session/prompt.py: Agent编排循环 (_setup → _loop → _post_loop)      │
│  session/processor.py: 单步处理器 (~1690行, 10阶段)                   │
│  session/tool_executor.py: 流式工具执行器 (并发+独占+兄弟中止)         │
│  session/microcompact.py: 零成本3层压缩 (微压缩/预算/折叠)             │
│  session/compaction.py: LLM摘要2阶段压缩 (prune → summarize)        │
│  session/middlewares/: 可组合中间件链 (3个中间件)                     │
│  streaming/manager.py: GenerationJob + StreamManager (SSE核心)       │
│  streaming/events.py: 22种SSE事件类型常量                             │
│  provider/: LLM提供者抽象 (7个实现) + 双层索引注册表                   │
│  tool/: 工具系统 (~30个工具)  agent/: 代理注册+权限引擎                │
│  mcp/: MCP协议集成  channels/: 外部通道适配 (10+平台)                 │
│  memory/: 工作区内存注入+异步刷新队列                                  │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、全链路数据流：从用户输入到内容显示

### 2.1 完整请求生命周期（含代码级调用路径）

```
用户输入文本/附件
    │
    ▼
[前端] ChatForm → useChat.sendMessage()
    │  ├─ beginSending(): 乐观更新用户气泡 (Zustand chat-store)
    │  ├─ POST /api/chat/prompt {session_id, text, model?, provider_id?,
    │  │     agent?, attachments[], permission_presets, permission_rules,
    │  │     reasoning?, workspace?, format?}
    │  ├─ 成功: startGeneration(stream_id, session_id) → useSSE(stream_id)
    │  └─ 失败回滚: finishGeneration() + 恢复输入框内容
    │
    ▼
[后端] chat.py: start_prompt()  (路由入口, ~207行)
    │  ├─ _ensure_image_attachments_supported() → 非视觉模型拒绝附件
    │  ├─ 生成ULID: session_id, stream_id
    │  ├─ StreamManager.create_job(stream_id, session_id)
    │  │   └─ GenerationJob(事件缓冲5000, 订阅者队列, abort_event,
    │  │                    _response_futures, artifact_cache)
    │  │   └─ job.interactive = True (启用权限ask阻塞)
    │  ├─ asyncio.create_task(_run_with_semaphore(sm, job, coro))
    │  │   ├─ sm._semaphore.acquire(timeout=30s) — 信号量保护
    │  │   └─ run_generation(job, request, ...) → SessionPrompt
    │  └─ task.add_done_callback(_on_task_done) — 未处理异常兜底
    │       └─ 发送 AGENT_ERROR SSE事件 (防止UI卡在generating)
    │
    │  返回 PromptResponse {stream_id, session_id}  (即fire-and-forget)
    │       │
    │       ▼
    │  SessionPrompt.run()
    │    ├─ _setup()  [详见 3.2 节]
    │    └─ _loop()   [详见 3.2 节]
    │         │
    │         ▼
    │    SessionProcessor.process() [单步, 详见 3.3 节]
    │      ├─ Phase 1: _stream_llm_with_retry()
    │      │   ├─ _build_stream_args() → (reasoning_extra, safe_max_tokens, exclude_tools)
    │      │   │   └─ exclude_tools: 每日搜索配额超限 → 排除web_search
    │      │   │                    OpenAI订阅提供者 → 排除web_search(使用原生搜索)
    │      │   ├─ _check_vision_blocked() → 非视觉模型+图片 → "stop"
    │      │   ├─ stream_llm(provider, model_id, messages, ...)
    │      │   │   └─ llm.py: 工具规格解析 + provider.stream_chat()
    │      │   ├─ chunk分发:
    │      │   │   ├─ "text-delta" → 累积 + SSE TEXT_DELTA
    │      │   │   ├─ "reasoning-delta" → 累积 + SSE REASONING_DELTA
    │      │   │   ├─ "tool-call" → _handle_tool_call_chunk()
    │      │   │   │   ├─ loop_detector.check() → 警告/阻断
    │      │   │   │   ├─ tool_registry.get() → 精确→小写→invalid回退
    │      │   │   │   ├─ evaluate(tool, pattern, merged_permissions)
    │      │   │   │   │   └─ "ask" → _ask_permission()
    │      │   │   │   │       └─ SSE PERMISSION_REQUEST → job.wait_for_response(300s)
    │      │   │   │   ├─ 持久化"running" Part → SSE TOOL_START
    │      │   │   │   └─ StreamingToolExecutor.submit()
    │      │   │   │       ├─ concurrent_safe → asyncio.create_task (立即执行)
    │      │   │   │       └─ exclusive → _exclusive_queue (流后顺序执行)
    │      │   │   ├─ "web-search-start/result" → 原生搜索处理
    │      │   │   ├─ "usage" → self.usage_data
    │      │   │   ├─ "finish" → self.finish_reason
    │      │   │   └─ "error" → SSE AGENT_ERROR + "stop"
    │      │   └─ 空输出重试 (最多2次)
    │      │
    │      ├─ Phase 2-3: 错误/空输出处理
    │      ├─ Phase 4: _persist_text_and_reasoning()
    │      ├─ Phase 5: _dispatch_tool_calls() → 收集+持久化+副作用
    │      │   └─ _finalize_one_tool_result(): 超时/错误/成功3路
    │      │       ├─ _apply_tool_side_effects(): quota/file/todo/artifact
    │      │       ├─ _build_tool_persist_output(): 提醒+中间件
    │      │       └─ _maybe_switch_agent(): 代理切换
    │      ├─ Phase 6-7: 成本计算+步进完成持久化
    │      ├─ Phase 8: _check_context_overflow() → should_compact(85%)
    │      ├─ Phase 9: middleware_chain.run_on_step_complete()
    │      └─ Phase 10: has_tool_calls → "continue" | "stop"
    │
    ▼
[后端] _loop() 结果处理
    │  ├─ "compact" → _handle_compact_result()
    │  │   ├─ L3: context_collapse() → 丢弃1/3最旧消息 + 边界标记
    │  │   ├─ L4: run_compaction() → 2阶段(prune+summarize)
    │  │   ├─ Todo恢复: 注入未完成TODO提醒
    │  │   └─ 连续失败3次 → AGENT_ERROR
    │  ├─ "stop" → _handle_stop_result()
    │  │   ├─ 长度续写守卫 (≤3次)
    │  │   ├─ 首轮工具提示 (3+附件)
    │  │   ├─ 未完成TODO续写
    │  │   └─ 空输出提示 (仅1次)
    │  └─ 硬上限: step > max_steps → 最多多跑1步
    │
    ▼
[后端] _post_loop()
    │  ├─ 删除空助手消息壳
    │  ├─ 持久化累计cost/token
    │  ├─ 自动标题 (首次交互, SSE TITLE_UPDATE 在 DONE 之前)
    │  ├─ 排队工作区内存刷新 (异步)
    │  └─ SSE DONE {session_id, finish_reason, total_cost}
    │
    ▼
[前端] SSE消费链路
    │  SSEClient → useSSE → chatStore → React重渲染
    │  ├─ ProgressiveBuffer: 60ms批量合并delta
    │  ├─ 22个事件处理器
    │  ├─ 安全网: 1.2s debounce + 8s硬超时 + 15s空闲恢复
    │  ├─ 可见性感知: 远程模式后台30s暂停
    │  └─ 去重: LRU Set(256) + eventSessionId guard
    │
    ▼
[前端] 消息渲染
    MessageList → AssistantMessageGroup → MessageContent
      → Part渲染器 (13种)
```

### 2.2 关键时序：发送消息到看到第一个token

| 阶段 | 耗时估计 | 关键操作 | 代码入口 |
|------|---------|---------|---------|
| 用户点击发送 → API请求发出 | ~50ms | 乐观更新用户气泡, fetch POST | chat-store.beginSending() |
| 后端接收 → 创建GenerationJob | ~10ms | 信号量获取(30s超时), 后台任务启动 | chat.py:start_prompt() |
| _setup() 完成 | ~200-500ms | Agent/Model解析, 系统提示组装, 权限5层合并, DB写入 | SessionPrompt._setup() |
| 历史消息加载+压缩 | ~100-300ms | DB查询, L1微压缩, L2工具预算, 中间件链 | _prepare_step_messages() |
| LLM首token延迟 (TTFT) | ~500-3000ms | provider.stream_chat() 首个chunk | SessionProcessor._stream_llm_with_retry() |
| SSE传输到前端 | ~10ms | GenerationJob.publish() → 订阅者asyncio.Queue | GenerationJob.publish() |
| ProgressiveBuffer刷新 | ≤60ms | 60ms批量合并delta, 防渲染卡顿 | ProgressiveBuffer.flushPending() |
| React重渲染 | ~16ms | Zustand set() → 组件更新 | chatStore.appendTextDelta() |

**端到端首token延迟**: ~1-4秒 (主要瓶颈在LLM TTFT)

## 三、后端核心链路深度分析

### 3.1 数据模型层

**Session** (`backend/app/models/session.py`):
- ULID主键, 关联Project (workspace持久化记录)
- `summary_additions/deletions/files/diffs`: 每步后的变更统计
- `permission: list[dict]`: 会话级权限覆盖 (JSON列)
- `time_compacting/time_archived`: 压缩/归档时间戳

**Message** (`backend/app/models/message.py`):
- ULID主键, 关联Session (CASCADE删除)
- `data: JSON`: 完整MessageInfo
  - 用户消息: `{role, model:{provider_id, model_id}, agent, system?, variant?, tools?}`
  - 助手消息: `{role, parent_id, agent, model_id, provider_id, cost, tokens, tokens_accumulated, error?, finish?}`

**Part** (`backend/app/models/message.py`):
- ULID主键, 反规范化 `session_id` (索引 `ix_part_session_id`)
- `data: JSON`: 按 `type` 字段区分的8种Part:
  - `text`: `{type, text, synthetic?}` — synthetic标记系统注入
  - `reasoning`: `{type, text}`
  - `tool`: `{type, tool, call_id, state:{status, input, output?, metadata?, title?, time_start?, time_end?, time_compacted?}}`
  - `step-start`: `{type, step, snapshot?}`
  - `step-finish`: `{type, reason, tokens, cost}`
  - `compaction`: `{type, auto}`
  - `subtask`: `{type, session_id, title, description}`
  - `file`: `{type, file_id, name, path, size, mime_type, source, content_hash}`

### 3.2 Agent编排循环 (`backend/app/session/prompt.py`, 972行)

`SessionPrompt` 是Agent循环的核心编排器，采用 **setup → loop → post-loop** 三阶段生命周期:

#### _setup() — 5步初始化

1. **解析Agent**: `agent_registry.get(request.agent)` → 默认代理兜底
2. **解析模型+提供者**: 请求级 > Agent级 > 首个免费模型 > 首个可用模型; 失败时自动refresh_models(); Anthropic启用提示缓存(`as_cached_blocks()`)
3. **创建/加载Session + 持久化用户消息+附件**: 创建session + user message + text part + file parts; FTS索引(含工作区外附件)
4. **构建系统提示**: 缓存段(Anthropic cache_control: agent.system_prompt + AGENTS.md) + 动态段(workspace_memory + skills + environment)
5. **5层权限合并**: GLOBAL_DEFAULTS → agent.permissions → preset_permissions → request_permissions → session_permissions; 重建artifact_cache(从历史消息恢复)

#### _loop() — 主循环

```python
while True:
    if abort_event: break
    step += 1
    if step > max_steps:
        if _should_break_on_hard_cap(): break  # 首次: 注入总结请求, 继续1步
    
    SSE STEP_START {step, session_id}
    llm_messages, mw_ctx = _prepare_step_messages()  # DB→sanitize→L1→L2→中间件
    _create_assistant_message_shell()
    
    processor = SessionProcessor(self, llm_messages, assistant_msg_id, mw_ctx)
    result = processor.process()  # "continue" | "stop" | "compact"
    _accumulate_step_metrics(processor)
    
    if result == "compact":  # L3折叠→L4摘要, 失败3次break
    if result == "stop":     # 4层守卫, 全通过break
```

#### _handle_stop_result() — 4层继续守卫

| 守卫 | 条件 | 动作 | 上限 |
|------|------|------|------|
| 长度续写 | `finish_reason == "length"` | 注入系统消息, 继续循环 | ≤3次 |
| 首轮工具提示 | `step==1 && 3+附件 && 无工具调用` | 注入工具使用提示 | 1次 |
| TODO未完成 | 存在pending/in_progress todo | 注入继续工作提醒 | max_continuation_attempts |
| 空输出提示 | 全部步骤无文本输出 | 注入总结请求(仅文本, 不用工具) | 1次 |

### 3.3 单步处理器 (`backend/app/session/processor.py`, ~1690行)

`SessionProcessor` 处理单步LLM调用+工具分派，分为10个阶段:

| 阶段 | 方法 | 功能 | 关键细节 |
|------|------|------|---------|
| 1 | `_init_step_state()` | 初始化累加器 | text/reasoning/tool_calls, StreamingToolExecutor, exec_metadata |
| 2 | `_persist_step_start()` | 持久化step-start Part | DB写入 |
| 3 | `_stream_llm_with_retry()` | LLM流式调用+重试 | 最多3次重试(指数退避), 空输出重试2次 |
| 4 | `_handle_stream_error()` | 流式错误处理 | 上下文溢出→"compact", 其他→错误完成 |
| 5 | `_handle_empty_output_after_retries()` | 空输出处理 | 删除空助手消息, 返回"continue" |
| 6 | `_persist_text_and_reasoning()` | 持久化文本+推理Parts | 判断has_text |
| 7 | `_dispatch_tool_calls()` | 工具结果收集+持久化 | 并发+独占, 超时/错误/成功3路 |
| 8 | `_compute_step_cost()` | 步进成本计算 | pricing.prompt * input + pricing.completion * output |
| 9 | `_persist_step_finish()` | 步进完成持久化 | SSE STEP_FINISH + DB step-finish Part |
| 10 | `_check_context_overflow()` | 上下文溢出检查 | should_compact() → 85%阈值 |

#### 工具调用分派详细流程

```
tool-call流块 → loop_detector.check() → tool_registry.get() → evaluate()
  │
  ├─ "deny" → SSE TOOL_ERROR + 持久化error Part
  ├─ "ask" (interactive) → SSE PERMISSION_REQUEST → wait_for_response(300s)
  │   ├─ allowed+remember → _remember_permission_rule() → 追加到session+merged
  │   └─ !allowed → SSE TOOL_ERROR + 持久化error Part
  │
  ├─ 持久化"running" Part → SSE TOOL_START
  ├─ 构建 ToolContext (含_publish_fn, _app_state, _model_id, _job, _depth)
  └─ StreamingToolExecutor.submit()
      ├─ is_concurrency_safe → asyncio.create_task (立即执行)
      └─ exclusive → _exclusive_queue (流后顺序执行)
```

#### 工具结果后处理 (_finalize_one_tool_result)

```
执行结果 → timed_out/error/成功 3路
  ├─ 成功 → SSE TOOL_RESULT + 持久化completed Part
  │   ├─ _apply_tool_side_effects(): quota/file/todo/artifact
  │   ├─ _build_tool_persist_output(): presentation_reminder + loop_warn + todo_remind + middleware
  │   ├─ 附件Parts持久化
  │   └─ _maybe_switch_agent(): plan工具代理切换 → rebuild_permissions_and_prompt()
```

### 3.4 工具执行引擎 (`backend/app/session/tool_executor.py`, 237行)

**并发策略**:
- **并发安全工具** (`is_concurrency_safe=True`): read, glob, grep, search, web_fetch等 — 在LLM流式传输期间立即启动
- **独占工具** (`is_concurrency_safe=False`): write, edit, bash等 — 流后顺序执行
- **兄弟中止**: bash失败(`SIBLING_ABORT_TOOLS`) → 取消所有并行兄弟任务

**collect()三步**: 1. 等待并发(含兄弟中止) → 2. 顺序执行独占 → 3. 按提交索引排序返回

### 3.5 Provider系统 (`backend/app/provider/`)

**BaseProvider ABC**: `id`, `list_models()`, `stream_chat()`, `health_check()`, `clear_cache()`(可选)

**ProviderRegistry** — 双层索引:
- `_model_index: dict[str, tuple]` — 快速查找(直接提供者优先于聚合器如OpenRouter)
- `_full_models: list[tuple]` — 完整列表(含不同提供者的同一模型)
- `refresh_models()`: 并发刷新所有提供者(45s超时/提供者)

**LLM流式桥接** (`session/llm.py`): 工具规格解析 → `provider.stream_chat()` → 规范化StreamChunk

### 3.6 SSE流式基础设施 (`backend/app/streaming/`)

#### GenerationJob 核心状态

| 属性 | 类型 | 用途 |
|------|------|------|
| `events` | `list[SSEEvent]` | 事件缓冲区(≤5000, 用于重连重放) |
| `subscribers` | `list[Queue]` | 多订阅者队列(每个SSE连接一个) |
| `_response_futures` | `dict[str, Future]` | 权限/问题阻塞(call_id → Future) |
| `_response_queue` | `Queue` | 响应早于等待的race condition兜底 |
| `interactive` | `bool` | True时启用权限ask阻塞 |
| `artifact_cache` | `dict` | 跨生成更新(identifier → content) |

**事件发布**: 单调ID → 缓冲区追加(溢出截断) → 分发到所有订阅者Queue → QueueFull时终端事件(DONE/AGENT_ERROR)强制送达, 非终端发送DESYNC

**重连重放**: `subscribe(last_event_id)` → 过滤重放 → 缓冲区溢出时截断+DESYNC → 已完成则立即发送None哨兵

**交互式阻塞**: `wait_for_response(call_id, 300s)` → Future等待; `submit_response()` → 设置Future结果或存入兜底Queue

#### 22种SSE事件类型

| 分类 | 事件 | 产生位置 |
|------|------|---------|
| 文本流 | text-delta, reasoning-delta | processor: LLM chunk |
| 工具流 | tool-call, tool-result, tool-error | processor: 工具生命周期 |
| 步进流 | step-start, step-finish | prompt._loop() / processor |
| 压缩流 | compacted, compaction-start/phase/progress/error | compaction |
| 交互流 | permission-request, question, plan-review, *-resolved | _ask_permission() / 工具 / chat.py |
| 控制流 | done, agent-error, retry, desync, title-update, model-loading | prompt / processor / GenerationJob |

### 3.7 压缩策略 (4层)

| 层级 | 策略 | LLM成本 | 实现 | 触发条件 |
|------|------|---------|------|---------|
| L1 | 微压缩 | 零 | 替换旧工具输出为存根(保护最近2轮, 阈值2000 tokens) | 每步自动 |
| L2 | 工具结果预算 | 零 | 总token上限100K, 超出替换最大项 | 每步自动 |
| L3 | 上下文折叠 | 零 | 丢弃最旧1/3消息, 插入合成边界(最小保留6条) | compact结果时 |
| L4 | LLM摘要 | 有 | compaction代理生成结构化摘要(Goal→Instructions→Discoveries→Accomplished→Files) | compact+L3不够时 |

**L4两阶段**: Phase1 prune(标记[truncated], 保护40K tokens) → Phase2 summarize(compaction代理, max_tokens=4096, 每200字符发COMPACTION_PROGRESS)

**触发条件**: `should_compact()` — `total_tokens >= min(usable_context, max_context * 0.85)`

### 3.8 权限引擎 (`backend/app/agent/permission.py`)

**5层"最后匹配胜出"模型**:
```
GLOBAL_DEFAULTS(allow*, ask bash/write/edit, deny question/plan, ask *.env)
  → Agent.permissions
  → presets_ruleset(file_changes→allow write+edit, run_commands→allow bash)
  → request_permissions
  → session_permissions
```

**2维匹配** (工具名 × 资源模式): `evaluate(permission, pattern, ruleset)` → 遍历所有规则, 最后匹配胜出 → `"allow"` | `"deny"` | `"ask"`

**权限请求交互**: 后端SSE PERMISSION_REQUEST(含脱敏参数) → 前端POST /chat/respond → `job.submit_response()` → `Future.set_result()`

### 3.9 中间件链 (`backend/app/session/middlewares/`)

3个中间件, 按顺序: DanglingToolCallMiddleware(修复悬空工具调用) → LoopDetectionMiddleware(循环检测) → TodoReminderMiddleware(TODO提醒)

### 3.10 MCP集成

`ConnectorRegistry → McpManager → McpClient(每服务器1个) + McpToolWrapper(适配MCP→ToolDefinition)`

**延迟工具发现**: `tool_search` 让模型按需发现MCP工具, 已发现的存储在 `SessionPrompt.discovered_tools`

## 六、潜在瓶颈与优化方向

### 6.1 性能瓶颈

| 瓶颈 | 位置 | 影响 | 优化方向 | ROI |
|------|------|------|---------|-----|
| **历史消息全量加载** | get_message_history_for_llm() + microcompact | 每步DB查询+转换+微压缩 | 增量缓存: 压缩锚点后增量追加 | 高 |
| **LLM TTFT** | Provider.stream_chat() | 1-4秒首token延迟 | 提示缓存(Anthropic已支持), 预测性预取 | 高 |
| **ProgressiveBuffer 60ms** | use-sse.ts | 首token增加60ms延迟 | 自适应: 首chunk立即刷新, 后续批量 | 中 |
| **单并发生成** | StreamManager._semaphore(默认1) | 同一Session串行 | 跨Session并行, 信号量按Session粒度 | 中 |
| **微压缩每步执行** | microcompact.py | 每步CPU开销(token估计+遍历) | 增量标记: 仅压缩新增内容 | 中 |

### 6.2 可靠性风险

| 风险 | 位置 | 严重度 | 描述 |
|------|------|--------|------|
| **SSE重连DESYNC** | streaming/manager.py | 高 | 缓冲区溢出(>5000事件)导致客户端丢失事件 |
| **use-sse定时器风暴** | hooks/use-sse.ts | 中 | 6+并发定时器, 快速导航时可能泄漏 |
| **DB恢复竞态** | use-sse.ts finishFromDB | 中 | 多异步步骤间状态可能变化 |
| **后端进程崩溃恢复** | backend.rs | 中 | 60s内3次重启上限, 超限需手动重启 |
| **chat-store step_finish去重** | stores/chat-store.ts | 中 | 快速step_finish下LRU Set(256)可能溢出 |

### 6.3 架构优化建议

#### 高优先级

1. **历史消息增量加载**: 当前每步全量 `get_message_history_for_llm()` + `microcompact_messages()` + `apply_tool_result_budget()`。可在压缩锚点后增量追加, 避免重复DB查询和格式转换。预计减少30-50%的步间延迟。

2. **ProgressiveBuffer自适应刷新**: 首chunk立即刷新(减少60ms首token延迟), 后续60ms批量(保持渲染性能)。

3. **processor.py拆分**: 1690行的单步处理器是最高复杂度文件。建议按阶段拆分为: stream_handler(LLM流式+重试)、tool_dispatcher(工具分派+持久化+副作用)、step_tracker(成本+上下文检查+步进生命周期)。

#### 中优先级

4. **前端SSE状态机**: use-sse.ts的837行和6+定时器是维护负担。引入状态机(空闲→连接中→活跃→步进中→重连中→断开), 替代分散定时器管理。

5. **SSE事件协议分组**: 22种事件按生命周期阶段分组(生成/工具/压缩/交互/控制), 减少处理器数量。

6. **自适应压缩触发**: 85%阈值固定, 可根据模型类型、对话模式动态调整。

#### 低优先级

7. **流式状态与持久状态统一**: Zustand+React Query双层架构增加复杂度, 但大规模重构ROI不高。

8. **Token管理统一**: 当前分散在3处, 可引入统一Token管理器。

### 3.2 Agent编排循环 (`backend/app/session/prompt.py`, 972行)

`SessionPrompt` 是Agent循环的核心编排器，采用 **setup → loop → post-loop** 三阶段生命周期:

#### _setup() — 5步初始化

1. **解析Agent**: `agent_registry.get(request.agent)` → 默认代理兜底
2. **解析模型+提供者**: 请求级 > Agent级 > 首个免费模型 > 首个可用模型; 失败时自动refresh_models(); Anthropic启用提示缓存(`as_cached_blocks()`)
3. **创建/加载Session + 持久化用户消息+附件**: 创建session + user message + text part + file parts; FTS索引(含工作区外附件)
4. **构建系统提示**: 缓存段(Anthropic cache_control: agent.system_prompt + AGENTS.md) + 动态段(workspace_memory + skills + environment)
5. **5层权限合并**: GLOBAL_DEFAULTS → agent.permissions → preset_permissions → request_permissions → session_permissions; 重建artifact_cache(从历史消息恢复)

#### _loop() — 主循环

```python
while True:
    if abort_event: break
    step += 1
    if step > max_steps:
        if _should_break_on_hard_cap(): break  # 首次: 注入总结请求, 继续1步
    
    SSE STEP_START {step, session_id}
    llm_messages, mw_ctx = _prepare_step_messages()  # DB→sanitize→L1→L2→中间件
    _create_assistant_message_shell()
    
    processor = SessionProcessor(self, llm_messages, assistant_msg_id, mw_ctx)
    result = processor.process()  # "continue" | "stop" | "compact"
    _accumulate_step_metrics(processor)
    
    if result == "compact":  # L3折叠→L4摘要, 失败3次break
    if result == "stop":     # 4层守卫, 全通过break
```

#### _handle_stop_result() — 4层继续守卫

| 守卫 | 条件 | 动作 | 上限 |
|------|------|------|------|
| 长度续写 | `finish_reason == "length"` | 注入系统消息, 继续循环 | ≤3次 |
| 首轮工具提示 | `step==1 && 3+附件 && 无工具调用` | 注入工具使用提示 | 1次 |
| TODO未完成 | 存在pending/in_progress todo | 注入继续工作提醒 | max_continuation_attempts |
| 空输出提示 | 全部步骤无文本输出 | 注入总结请求(仅文本, 不用工具) | 1次 |

### 3.4 工具执行引擎 (`backend/app/session/tool_executor.py`, 237行)

### 3.7 压缩策略 (4层)

| 层级 | 策略 | LLM成本 | 实现 | 触发条件 |
|------|------|---------|------|---------|

### 5.1 进程管理 (`backend.rs`, 875行)

**启动**: 选端口 → 定二进制 → 启动子进程 → 健康检查(60次×500ms) → Token加载(0600文件, 5s) → `backend-ready` → 看门狗

**看门狗**: 每10s轮询/livez → 3次失败 → 杀进程+自动重启(指数退避1s→2s→4s, 60s内3次上限)

**优雅关闭**: SIGTERM → 等12s → SIGKILL

### 5.2 桌面 vs Web模式差异

| 关注点 | 桌面模式 | Web开发模式 |
|--------|---------|------------|
| 后端URL | IPC获取 `http://127.0.0.1:{port}` | `NEXT_PUBLIC_API_URL` |
| 认证 | Bearer token (0600文件) | 无 |
| SSE认证 | `?token=` 查询参数 | 无 |
| SSE连接 | EventSource (GET) | EventSource (GET) |
| 文件上传 | Tauri原生对话框 + IPC | Web文件选择器 |

## 六、潜在瓶颈与优化方向

### 6.1 性能瓶颈

| 瓶颈 | 位置 | 影响 | 优化方向 | ROI |
|------|------|------|---------|-----|
| **历史消息全量加载** | get_message_history_for_llm() + microcompact | 每步DB查询+转换+微压缩 | 增量缓存: 压缩锚点后增量追加 | 高 |
| **LLM TTFT** | Provider.stream_chat() | 1-4秒首token延迟 | 提示缓存(Anthropic已支持), 预测性预取 | 高 |
| **ProgressiveBuffer 60ms** | use-sse.ts | 首token增加60ms延迟 | 自适应: 首chunk立即刷新, 后续批量 | 中 |
| **单并发生成** | StreamManager._semaphore(默认1) | 同一Session串行 | 跨Session并行, 信号量按Session粒度 | 中 |
| **微压缩每步执行** | microcompact.py | 每步CPU开销(token估计+遍历) | 增量标记: 仅压缩新增内容 | 中 |

### 6.2 可靠性风险

| 风险 | 位置 | 严重度 | 描述 |
|------|------|--------|------|
| **SSE重连DESYNC** | streaming/manager.py | 高 | 缓冲区溢出(>5000事件)导致客户端丢失事件 |
| **use-sse定时器风暴** | hooks/use-sse.ts | 中 | 6+并发定时器, 快速导航时可能泄漏 |
| **DB恢复竞态** | use-sse.ts finishFromDB | 中 | 多异步步骤间状态可能变化 |
| **后端进程崩溃恢复** | backend.rs | 中 | 60s内3次重启上限, 超限需手动重启 |
| **chat-store step_finish去重** | stores/chat-store.ts | 中 | 快速step_finish下LRU Set(256)可能溢出 |

### 6.3 架构优化建议

#### 高优先级

1. **历史消息增量加载**: 当前每步全量 `get_message_history_for_llm()` + `microcompact_messages()` + `apply_tool_result_budget()`。可在压缩锚点后增量追加, 避免重复DB查询和格式转换。预计减少30-50%的步间延迟。

2. **ProgressiveBuffer自适应刷新**: 首chunk立即刷新(减少60ms首token延迟), 后续60ms批量(保持渲染性能)。

3. **processor.py拆分**: 1690行的单步处理器是最高复杂度文件。建议按阶段拆分为: stream_handler(LLM流式+重试)、tool_dispatcher(工具分派+持久化+副作用)、step_tracker(成本+上下文检查+步进生命周期)。

#### 中优先级

4. **前端SSE状态机**: use-sse.ts的837行和6+定时器是维护负担。引入状态机(空闲→连接中→活跃→步进中→重连中→断开), 替代分散定时器管理。

5. **SSE事件协议分组**: 22种事件按生命周期阶段分组(生成/工具/压缩/交互/控制), 减少处理器数量。

6. **自适应压缩触发**: 85%阈值固定, 可根据模型类型、对话模式动态调整。

#### 低优先级

7. **流式状态与持久状态统一**: Zustand+React Query双层架构增加复杂度, 但大规模重构ROI不高。

8. **Token管理统一**: 当前分散在3处, 可引入统一Token管理器。

## 九、核心发现总结

### 9.1 架构亮点

1. **职责分离清晰**: 三层架构(Tauri壳→Python后端→React前端)通过ADR-0001确保AI逻辑不泄漏到壳层, 所有Agent/Tool/Provider逻辑集中在Python后端
2. **流式架构成熟**: SSE + 事件缓冲(5000) + 重连重放 + ProgressiveBuffer(60ms) + 多安全网(1.2s/8s/15s), 形成完善的可靠性保障
3. **压缩策略层次化**: 4层从零成本到LLM摘要, L1/L2每步自动执行无感, L3/L4按需触发, 渐进式压缩最大化token利用率
4. **工具执行并发化**: StreamingToolExecutor实现LLM流中并行(安全工具立即启动) + 流后独占顺序执行 + bash兄弟中止, 与Claude Code架构对齐
5. **权限模型精细化**: 5层合并+2维匹配(工具名×资源模式)+交互式ask阻塞+敏感参数脱敏+会话级记忆, 安全性高且用户友好
6. **前端可靠性纵深**: 三层安全网(1.2s debounce/8s硬超时/15s空闲恢复) + DB恢复 + DESYNC容错 + 可见性感知, 保障弱网和异常场景下的用户体验

### 9.2 最大优化杠杆

1. **历史消息增量加载** (ROI最高): 当前每步全量DB查询+微压缩, 是步间延迟的主要来源。增量缓存可减少30-50%步间延迟
2. **ProgressiveBuffer自适应** (ROI高): 首chunk立即刷新可减少60ms首token感知延迟, 实现成本极低
3. **processor.py拆分** (ROI中): 1690行单文件是理解和维护瓶颈, 拆分后可独立演进和测试

### 9.3 关键风险

1. **SSE重连DESYNC**: 缓冲区5000事件上限, 长时间生成(如代码重构)可能溢出导致前端丢失事件
2. **use-sse定时器风暴**: 6+并发定时器在快速导航场景下易泄漏, 需状态机化
3. **DB恢复竞态**: finishFromDatabase()涉及多个异步步骤, 状态可能在步骤间变化
